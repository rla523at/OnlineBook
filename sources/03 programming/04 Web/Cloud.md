# Cloud

## 서버란 무엇인가

서버는 다른 컴퓨터의 요청을 받아 필요한 일을 처리하고 결과를 돌려주는 컴퓨터 또는 프로그램이다.

- 사용자의 노트북이나 휴대폰은 보통 client다.
- 서버는 요청을 받아 처리하는 쪽이다.

예:

```text
사용자가 쇼핑몰 앱을 연다.
  -> 앱이 서버에 상품 목록을 요청한다.
  -> 서버가 DB에서 상품 정보를 읽는다.
  -> 서버가 응답을 보낸다.
  -> 앱 화면에 상품이 표시된다.
```

서버는 역할의 개념이다. 어떤 컴퓨터든 남의 요청을 받아 처리하도록 구성하면 서버가 될 수 있다.

## 서버를 직접 운영한다는 뜻

회사가 직접 서버를 운영한다면 다음을 모두 책임져야 한다.

- 물리 서버 구매
- 데이터센터 또는 서버실 준비
- 전원, 냉각, 네트워크 구성
- 운영체제 설치
- 보안 설정
- 모니터링과 장애 대응
- 고장 시 부품 교체
- 트래픽 증가 시 서버 증설

즉 서버 운영은 컴퓨터 한 대를 사는 문제가 아니라, 24시간 안정적으로 서비스를 제공하기 위한 물리 환경과 운영 체계를 관리하는 일이다.

## 클라우드 컴퓨팅

클라우드는 물리 서버를 클라우드 사업자가 미리 준비해 두고, 사용자가 그 자원을 필요한 만큼 빌려 쓰는 방식이다.

여기서 빌린다는 말은 장비를 집으로 가져오는 것이 아니라, 인터넷을 통해 내 계정 안에 서버 1대가 생긴 것처럼 사용할 권한을 받는다는 뜻이다.

클라우드 서버를 만들면 보통 다음을 할 수 있다.

- OS가 설치된 가상 서버 생성
- CPU, 메모리, 디스크, 네트워크 할당
- SSH로 서버 접속
- 웹 서버와 애플리케이션 설치
- 사용량 또는 실행 시간 기준 비용 지불
- 필요 없을 때 중지 또는 삭제

비유하면 다음과 같다.

- 서버 직접 구매
  - 건물을 직접 사서 전기, 수도, 보안까지 직접 관리
- 클라우드 서버 사용
  - 호텔 방을 필요할 때 빌리고 시설 관리는 호텔이 담당

## EC2 같은 가상 서버

EC2 같은 서비스는 가상 서버 1대를 빌려 쓰는 서비스다.

사용자는 다음을 직접 관리한다.

- 서버 생성
- SSH 접속
- 패키지 설치
- 애플리케이션 배포
- 로그 확인
- 보안 업데이트
- 장애 대응

핵심은 EC2가 서버 자체를 제공하지만, 그 서버에서 무엇을 어떻게 운영할지는 사용자가 책임진다는 점이다.

## 관리형 배포 서비스

Elastic Beanstalk 같은 관리형 배포 서비스는 애플리케이션 배포와 운영에 필요한 주변 구성을 일부 자동화한다.

예:

- 서버 생성
- 로드 밸런서 구성
- 오토스케일링
- 기본 모니터링
- 배포 절차 자동화

관계는 다음처럼 이해하면 된다.

```text
EC2
  -> 가상 서버 자체

Elastic Beanstalk 같은 서비스
  -> EC2와 주변 운영 구성을 묶어 애플리케이션 배포를 쉽게 해 주는 상위 서비스
```

관리형 서비스는 편하지만 내부 구조를 모르면 문제가 생겼을 때 원인을 찾기 어렵다. 그래서 웹 서버, process manager, reverse proxy, DB connection 같은 기본 개념은 여전히 이해해야 한다.

## 클라우드와 웹 배포

클라우드 서버 위에 직접 웹 애플리케이션을 올리면 보통 다음 구성 요소가 필요하다.

```text
Browser
  -> HTTPS
  -> Nginx
  -> frontend static files or backend API
  -> FastAPI backend
  -> PostgreSQL
```

운영 관점에서는 다음도 함께 관리한다.

- SSH 접근
- 운영자 계정과 service account
- systemd service
- Nginx 설정
- TLS 인증서
- DB migration
- 로그와 모니터링
- backup과 restore

이런 세부 배포 구조는 [WebDeployment.md](./WebDeployment.md)를 참고한다.

## 자주 하는 실수

- 서버를 단순히 컴퓨터 한 대로만 생각하고 운영 책임을 과소평가하는 경우
- 클라우드 서버를 만들면 보안과 배포가 자동으로 해결된다고 생각하는 경우
- EC2 같은 가상 서버와 Elastic Beanstalk 같은 관리형 서비스를 같은 계층으로 보는 경우
- 배포 자동화 도구를 쓰면서도 내부 Nginx, systemd, DB 구조를 이해하지 않는 경우
